**版本**：v1.0 | **日期**：2026-04-08

**配套文档**：[业务配置后台设计文档-前端.md](./业务配置后台设计文档-前端.md)

## 1. 定位与范围

### 1.1 定位

业务配置后台（Metadata Admin）是智能对话网关的**运营控制台**。目标用户为业务线运营人员，提供表单化界面对客服 SOP 流程进行增删改查，**无需改代码即可生效**。

### 1.2 本周交付范围

| 功能 | 是否本周交付 |
|------|------------|
| SOP 流程 CRUD 接口 | ✅ |
| Active Intents 查询接口（供 Dify 拉取） | ✅ |
| Redis 缓存 + 写入时主动失效 | ✅ |
| 预置首批 4 条业务 SOP | ✅ |
| 前端管理界面 | ❌（本周仅后端接口，前端可用 Swagger UI 或 Postman 替代） |

## 2. 数据模型

### 2.1 `sop_definitions` 表

```sql
-- SOP metadata table: business staff configure flows here, no code change needed
CREATE TABLE sop_definitions (
    id              BIGSERIAL PRIMARY KEY,
    intent_key      VARCHAR(64)   NOT NULL UNIQUE, -- intent identifier, e.g. "verify_identity"
    intent_label    VARCHAR(128)  NOT NULL,         -- display name, e.g. "身份认证"
    allow_interrupt BOOLEAN       NOT NULL DEFAULT TRUE,  -- can this flow be interrupted?
    is_global       BOOLEAN       NOT NULL DEFAULT FALSE, -- global intent (answers anytime)?
    steps_config    JSONB         NOT NULL,               -- step pipeline definition (see 2.2)
    active_intents  TEXT[],       -- sub-intents this flow may trigger (injected into Dify)
    status          SMALLINT      NOT NULL DEFAULT 1,     -- 1=enabled, 0=disabled
    created_by      VARCHAR(64),
    created_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW(),
    updated_at      TIMESTAMPTZ   NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_sop_definitions_intent_key ON sop_definitions(intent_key);
CREATE INDEX idx_sop_definitions_status     ON sop_definitions(status);
```

### 2.2 `steps_config` 字段格式

`steps_config` 是一个有序步骤数组，网关按索引驱动对话前进。

```json
[
  {
    "step": 0,
    "description": "收集用户证件号",
    "required_slots": ["id_number"],
    "api_endpoint": null
  },
  {
    "step": 1,
    "description": "调用身份核验接口",
    "required_slots": [],
    "api_endpoint": "http://internal-api/verify_identity"
  }
]
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `step` | int | ✅ | 步骤索引，从 0 开始，必须连续 |
| `description` | string | ✅ | 步骤描述（运营可读，网关不使用） |
| `required_slots` | string[] | ✅ | 本步骤必须收集的槽位名，空数组表示无需收集 |
| `api_endpoint` | string\|null | ✅ | 槽位收集完毕后调用的内部业务 API URL，null 表示纯收集步骤 |

**约束**：
- `step` 索引必须从 0 连续递增，不可跳号
- 至少有 1 个步骤
- `api_endpoint` 若非 null，网关会在本步骤所有 `required_slots` 收集完毕后自动调用

## 3. 接口设计

### 3.1 接口列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/v1/admin/sop` | 查询所有 SOP（支持过滤） |
| `GET` | `/api/v1/admin/sop/{intent_key}` | 查询单条 SOP |
| `POST` | `/api/v1/admin/sop` | 新建 SOP |
| `PUT` | `/api/v1/admin/sop/{intent_key}` | 更新 SOP（全量替换） |
| `DELETE` | `/api/v1/admin/sop/{intent_key}` | 软删除（status=0） |
| `GET` | `/api/v1/dify/active_intents` | 查询当前激活意图列表（供 Dify 拉取） |

### 3.2 `GET /api/v1/admin/sop`

**Query 参数**

| 参数 | 类型 | 说明 |
|------|------|------|
| `status` | int | 过滤状态，1=启用，0=禁用，不传返回全部 |

**响应示例**

```json
{
  "data": [
    {
      "intent_key": "verify_identity",
      "intent_label": "身份认证",
      "allow_interrupt": true,
      "is_global": false,
      "steps_config": [...],
      "active_intents": ["verify_identity"],
      "status": 1,
      "created_at": "2026-04-08T10:00:00Z",
      "updated_at": "2026-04-08T10:00:00Z"
    }
  ],
  "total": 4
}
```

### 3.3 `POST /api/v1/admin/sop`

**请求体**

```json
{
  "intent_key": "query_promotion",
  "intent_label": "优惠活动查询",
  "allow_interrupt": true,
  "is_global": false,
  "steps_config": [
    {
      "step": 0,
      "description": "调用活动查询接口",
      "required_slots": [],
      "api_endpoint": "http://internal-api/query_promotion"
    }
  ],
  "active_intents": ["query_promotion"],
  "created_by": "ops_alice"
}
```

**成功响应** `201 Created`

```json
{ "intent_key": "query_promotion", "status": 1 }
```

**失败响应** `409 Conflict`（intent_key 已存在）

```json
{ "error": "intent_key 'query_promotion' already exists" }
```

### 3.4 `PUT /api/v1/admin/sop/{intent_key}`

全量替换，请求体与 POST 相同（不含 `intent_key`）。
写入成功后**立即使 Redis 缓存失效**（见第 4 节）。

**成功响应** `200 OK`

```json
{ "intent_key": "query_promotion", "updated_at": "2026-04-08T11:00:00Z" }
```

### 3.5 `DELETE /api/v1/admin/sop/{intent_key}`

软删除：将 `status` 置为 0，不物理删除。
写入后同步失效 Redis 缓存。

**成功响应** `200 OK`

```json
{ "intent_key": "query_promotion", "status": 0 }
```

### 3.6 `GET /api/v1/dify/active_intents`

Dify Start Node 在每次调用前拉取，获取当前所有启用的意图键名，用于注入 LLM System Prompt 防止意图幻觉。

**响应示例**

```json
{
  "intents": [
    { "key": "verify_identity",  "label": "身份认证" },
    { "key": "query_promotion",  "label": "优惠活动查询" },
    { "key": "query_recharge",   "label": "充值查询" },
    { "key": "query_withdrawal", "label": "提现查询" }
  ]
}
```

## 4. 缓存策略

```
Redis Key: sop:active_intents
类型:      String（JSON 序列化）
TTL:       300 秒（5 分钟）
```

**失效时机**：POST / PUT / DELETE 任意写操作成功后，网关**同步执行** `redis.delete("sop:active_intents")`，下次读取时重建。

**读取优先级**：
1. 命中 Redis 缓存 → 直接返回
2. 未命中 → 查询 `sop_definitions WHERE status=1` → 写入 Redis → 返回

---

## 5. 校验规则

| 字段 | 规则 |
|------|------|
| `intent_key` | 仅允许小写字母、数字、下划线；长度 2-64 |
| `intent_label` | 非空字符串，长度 1-128 |
| `steps_config` | 数组非空；`step` 从 0 连续递增；每个 `step` 对象必须包含所有必填字段 |
| `api_endpoint` | 若非 null，必须是合法 URL（以 `http://` 或 `https://` 开头） |

校验失败统一返回 `422 Unprocessable Entity`：

```json
{ "error": "steps_config[1].step must be 1, got 2 (non-consecutive)" }
```

## 8. 边界说明

- **`api_endpoint` 的值本周不验证连通性**，网关运行时调用；业务部门填写时请参照内部 API 文档
- **`required_slots` 的槽位名**由 Dify 侧 System Prompt 决定语义，本后台只负责存储字符串，两侧需手工保持一致
- **前端界面**不在本周范围内，运营人员本周通过 Swagger UI（`/docs`）或直接 POST 数据录入
