## 产品说明书：任务驱动型动态 AI 客服系统 (V1.0)

### 一、 产品定位与愿景

本系统是一款**强业务流程导向（Task-Oriented）**的企业级 AI 客服中间件。有别于传统的闲聊机器人，本系统的核心使命是“按标准化 SOP 解决客户问题”。通过将大模型的“语义理解能力”与传统代码的“状态机确切性”相解耦，实现业务线无代码动态配置、多轮复杂交互不跑偏、且支持全渠道无缝人工接管的终极客服闭环。

### 二、 全局系统架构

系统采用”控制面与数据面分离”的架构，共包含 4 个核心模块：

1. **统一接入层 (Chatwoot)：** 负责多渠道（**Line、WhatsApp**，未来扩展 Web/App）的消息统一接入与格式归一化，兼顾人工客服工作台功能。
2. **调度控制中枢 (Core Gateway - 核心网关)：** 系统的”大脑”与业务引擎。完全掌控业务蓝图、对话栈、步骤游标，负责串联所有系统。
3. **语义翻译引擎 (Dify)：** 系统的”嘴巴”。维护唯一一条通用工作流，仅负责将用户的自然语言转化为网关可读的结构化 JSON，以及将系统数据润色为人话。
4. **元数据与审计中心 (PostgreSQL)：** 存储业务人员动态配置的 SOP 流程（无需改代码即可生效）及完整的结构化对话审计日志。

### 三、 核心功能模块详细说明

#### 3.1 极简业务配置后台 (Metadata Admin)

- **目标用户：** 业务线运营人员。
- **功能描述：** 提供类似表单的交互界面，允许业务人员动态增删改查客服 SOP。
- **核心配置项：**
  - **意图名称：** 触发该流程的用户意图（如：修改密码、查询航班）。
  - **插播策略：** `allow_interrupt`（是否允许当前流程被打断）、`is_global`（是否为全局插播意图，可随时解答）。
  - **多步执行管线 (Pipeline)：** 业务节点序列，定义该流程分为几步，每步收集什么参数，调用哪个内部业务 API（从预设的 API 列表中选择）。

#### 3.2 Dify 动态语义引擎

- **功能描述：** 处理语言多义性与上下文指代。
- **节点设计要求：**
  - **动态意图注入：** 每次识别前，先从网关拉取当前最新的 `Active Intents` 列表，防止意图膨胀与幻觉。
  - **状态游标读取：** 必须读取网关下发的 `inputs` 中的 `current_step`、`slots`、`stack_depth`、`pending_options`，结合当前意图状态精准提取用户提供的值。当 `pending_options` 非空时，代表系统正处于”等待选择”态，Dify 须将用户输入（如”第二个”）映射到 `pending_options` 中对应条目的 `id`（如 `G102`），再以 `action=advance` 返回。

#### 3.3 核心业务网关 (The Gateway) - [重点]

网关是整个系统的灵魂，必须具备以下三个核心机制：

##### 机制 A：多步状态机游标 (Step Chaining)

- 网关将复杂的业务流程拆分为单步执行。
- 通过在返回给 Dify 的负载中注入 `current_step`，强行牵引 Dify 进入下一步，实现跨多轮对话的流程串联。
- 当槽位未填满时，Dify 返回 `action=stay`，网关继续引导用户补充信息。

##### 机制 B：对话栈与插播恢复 (Dialogue Stack)

- 当用户在流程中途提出新意图时，网关根据业务配置校验打断合法性。
- 若合法，将当前任务（含已收集参数和游标）**压栈 (Push)**。
- 执行插播任务后，**弹栈 (Pop)** 恢复原任务，并组装特定的恢复话术（如：“...对了，您刚才正在...”）下发给 Dify。

##### 机制 C：全生命周期审计留痕 (Audit Logging)

- 网关拦截所有 Dify 解析出的 JSON 结果与 API 调用结果。
- 将 `[用户ID, 时间, 识别意图, 提取参数, 触发接口, 响应结果]` 写入数据库，支持内容运营人员按维度回溯排查。

#### 3.4 人工客服无缝流转 (Human Handoff)

- **触发机制：** 识别到用户转人工意图，或触发死循环防卫机制（重试 > 3次失败）。
- **流转路径：** 网关调用 Chatwoot API 分配工单，并将用户状态标记为 `is_human_mode: true`。
- **消息劫持：** 状态变更后，网关直接拦截用户消息，不再转发至 Dify，变为 Chatwoot 与终端用户的纯透明代理。客服在 Chatwoot 中点击”解决”后，状态复原，AI 重新接管。

### 四、 核心数据流转标准协议 (Gateway Payload)

网关与 Dify 之间的通讯必须遵循严格的 JSON 契约（**唯一版本，以下为事实源**）：

#### **4.1 网关下发给 Dify 的标准载荷（`inputs`）**

```json
{
  "user_message":  "用户当前消息",
  "conv_id":       "会话唯一标识",
  "channel":       "line | whatsapp",
  "current_step":  2,
  "total_steps":   5,
  "slots":           "{\"已收集槽位 JSON 字符串\"}",
  "stack_depth":     0,
  "pending_options": "[{\"id\":\"G101\",\"desc\":\"选项一\"},{\"id\":\"G102\",\"desc\":\"选项二\"}]",
  "history":         "[{\"role\":\"user\",\"content\":\"...\"}]"
}
```

> `slots` 和 `history` 以 **JSON 字符串**传入，避免 Dify 对嵌套对象处理不稳定。

#### **4.2 Dify 上报给网关的标准载荷（`outputs`）**

```json
{
  "action":      "advance | stay | interrupt | resume | handoff | complete | error",
  "next_step":   3,
  "reply_text":  "回复给用户的话（Dify 负责润色为自然语言）",
  "slots_patch": "{\"新增或更新的槽位 JSON 字符串\"}",
  "reason":      "内部推理摘要，不展示给用户"
}
```

**`action` 枚举语义**

| action | 含义 | 网关行为 |
|--------|------|---------|
| `advance` | 当前步骤完成，推进 | `current_step = next_step` |
| `stay` | 槽位未填满，继续等待 | 不更新步骤，回复引导话术 |
| `interrupt` | 用户偏离主流程 | 当前任务压栈，处理插播意图 |
| `resume` | 插播问题已处理完 | 弹栈恢复主流程，组装恢复话术 |
| `handoff` | 需要人工介入 | 触发人工转接流程（见第三节） |
| `complete` | 所有步骤执行完毕 | 弹栈、写审计、关闭会话 |
| `error` | Dify 无法处理 | 触发降级，重试 > 3 次转人工 |
