## 工期估算说明

| 阶段 | 详细分解 |
|------|---------|
| **Phase 0**（正常 1 周 / 冲刺 3 天） | 代码仓库 + 规范（0.5d）· PostgreSQL + Redis 部署（1d）· Docker Compose（1d）· Chatwoot 实例接入（1d）· Dify Cloud 申请（0.5d）· 环境变量配置（1d） |
| **Phase 1**（正常 1.5 周 / 冲刺 5 天） | 数据库 Schema + Alembic 迁移（2d）· Redis 会话管理（2d）· FastAPI 骨架 + 健康检查（1d）· 状态机 + 步骤链 + 对话栈（3d）· 审计日志（1d）· 单元测试（2d） |
| **Phase 2**（正常 1 周 / 冲刺 4 天） | Chatwoot API 封装（1.5d）· Webhook 签名校验（1d）· 消息过滤 `is_user_message`（0.5d）· 人工转接（2d）· Chatwoot Webhook 联调（1d） |
| **Phase 3**（正常 1 周 / 冲刺 4 天） | Dify API 封装（1d）· Dify Cloud Workflow 配置（2d）· Prompt 调优 + Code Node（2.5d）· 网关 ↔ Dify 联调（0.5d） |
| **Phase 4**（正常 1 周 / 冲刺 3 天） | R1 Webhook 超时加固（1d）· R2 Dify JSON 加固 + 压测（2d）· R3 死循环测试（0.5d）· R4 Redis Cold Restore（0.5d）· Prometheus 监控 + 告警（2d） |
| **Phase 5**（正常 1 周 / 冲刺 4 天） | E2E 测试（2d）· 安全审计（1d）· Dockerfile / 部署脚本（1.5d）· CI/CD 流水线（1d）· 生产部署 + 上线检查（0.5d） |

> d = 工作日（人天），所有工期为 2 人并行的估算。

## 一个月可行性评估

**结论：可完成，但需要接受以下取舍**

| 取舍项 | 说明 |
|--------|------|
| Phase 4 监控部分压缩 | Prometheus 告警可延至上线后 1 周内补齐，核心风险加固（R1-R4）必须做 |
| Phase 5 测试精简 | E2E 覆盖主要路径（正常流程、中断恢复、转人工），边界测试延至迭代补充 |
| Chatwoot 接入准备 | 使用现有实例，节省 Phase 0 中 Chatwoot 部署时间 |
| Dify Cloud | 无需自建，节省 Phase 0 中 Dify 部署时间 |
| 第一业务场景聚焦 | 身份认证、优惠活动、充值/提现查询场景相对轻量，Prompt 工程量可控 |

**冲刺前提条件**：
1. Phase 0 在 **3 天内完成**，不阻塞 Phase 1
2. Phase 2/3 必须 **完全并行**，各专注一个人
3. 每日站会同步进度，及时发现阻塞

## Phase 1：核心网关 + 数据库

### 1.1 数据库设计

- [ ] 编写 `001_init_schema.sql`，包含 `sop_definitions`、`biz_workflows`、`audit_logs` 表
- [ ] 配置数据库索引（conv_id、status、intent_key、created_at）
- [ ] 配置数据库迁移工具（Alembic）
- [ ] 编写数据库单元测试（表结构、索引）

### 1.2 SOP 元数据管理

- [ ] 实现 `sop_definitions` 的 CRUD API（`GET/POST/PUT/DELETE /api/v1/admin/sop`）
- [ ] 实现 `GET /api/v1/dify/active_intents`（供 Dify 拉取当前激活意图列表）
- [ ] 实现 SOP 配置缓存：Redis `sop:active_intents`，TTL 5 分钟，写入时主动失效
- [ ] 预置首批业务 SOP：身份认证、优惠活动查询、充值查询、提现查询

### 1.2 Redis 会话管理

- [ ] 实现 Redis 连接封装（连接池、重连）
- [ ] 实现会话读写：`session:{conv_id}`（Hash，TTL 24h）
- [ ] 实现幂等去重：`dedup:{message_id}`（String，TTL 5min）
- [ ] 实现分布式锁：`lock:conv:{conv_id}`（String，TTL 30s）
- [ ] 实现 Cold Restore：Redis cache miss 时从 DB 重建会话，`workflow_to_session()` 须包含 `is_human_mode` 和 `pending_options` 字段映射

### 1.3 网关骨架

- [ ] 搭建 FastAPI 项目骨架
- [ ] 实现 `/gateway/health` 健康检查接口
- [ ] 实现 Webhook 入口（`POST /gateway/webhook`），**500ms 内返回 202**
- [ ] 实现异步任务队列接入（Celery），业务逻辑异步执行，配置 DLQ

### 1.4 核心引擎

- [ ] 实现状态机：`status ∈ {init, in_progress, interrupted, handoff, complete, error}`
- [ ] 实现步骤链推进逻辑（advance、stay）
- [ ] 实现对话栈压栈 / 弹栈逻辑（interrupt、resume）
- [ ] 实现状态持久化：Redis 写入 + DB 同步更新

### 1.5 审计日志

- [ ] 实现 `audit_logs` 写入（append-only）
- [ ] 覆盖所有事件类型：`msg_received`、`dify_called`、`step_advanced` 等

### 1.6 Phase 1 交付物

- 数据库迁移脚本
- 核心网关可独立运行（Mock Dify 和 Chatwoot）
- 单元测试覆盖核心逻辑（状态机、对话栈）

## Phase 2：Chatwoot 集成

### 2.1 Chatwoot API 封装

- [ ] 实现 Chatwoot API Client（发送消息、分配坐席、获取坐席列表）
- [ ] 实现 Webhook 签名校验（`X-Chatwoot-Hmac-Sha256`）
- [ ] 实现消息过滤：`is_user_message()`，过滤 `message_type != 0` 和 `agent_bot`

### 2.2 人工转接完整闭环

- [ ] 实现 `handoff` action → `is_human_mode=true` + Chatwoot assign conversation
- [ ] 实现 Human Mode 消息旁路：`is_human_mode=true` 时不走 Dify，直接转发给 Chatwoot
- [ ] 实现坐席解决事件监听：订阅 Chatwoot `conversation_resolved` Webhook
- [ ] 实现 AI 恢复：`conversation_resolved` 后清除 `is_human_mode`，重置 status，发送恢复提示
- [ ] 实现转接上下文摘要（私信发送给坐席）
- [ ] 实现坐席选择策略（Round-robin，可扩展）

### 2.3 联调 Chatwoot

- [ ] 配置 Chatwoot Webhook（订阅 `message_created` + `conversation_resolved` 事件）
- [ ] 验证 Webhook 接收 + 回复闭环
- [ ] 验证坐席解决后 AI 恢复闭环（Human Mode → conversation_resolved → is_human_mode 清除）

### 2.4 Phase 2 交付物

- Chatwoot Webhook 联调通过
- 人工转接功能可演示
